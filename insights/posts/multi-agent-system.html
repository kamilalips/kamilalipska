I Built My First Multi-Agent System. Nobody Warned Me About This.

Everyone talks about agentic AI like it's a clean architecture diagram. Five boxes. Arrows between them. Ship it.

The reality looks different. Agents talk past each other. They write to the wrong fields. They loop forever on work no one told them to touch. And at some point you're staring at a pipeline that's technically running but producing nothing, with no error to debug.

This is what I learned building my first multi-agent content pipeline, from scratch, no frameworks, just the Claude API, Python, and a growing list of things that didn't work the way I expected.

It starts simple. Then it doesn't.

The idea was straightforward: automate the production of GEO- and SEO-optimized blog content. One agent researches topics. One writes. One edits. One handles links. One orchestrates the whole thing.

Each agent has a clearly defined role. They share a common state layer. Hand off work via status updates. In theory, the writer writes, the editor edits, and nothing steps on anything else.

That theory held for about two days.

What I didn't anticipate was how much complexity lives in the handoffs, not in the individual agents, but in what happens between them. Which fields does the editor read? When the linker finds a content problem and needs to escalate back to the writer, where does it write that feedback? If you haven't mapped this explicitly before you build, you will debug it later. Probably at 11pm.

The lesson I'd give anyone starting out: before any agent writes a line of code, write down every piece of shared state, who reads it, and who writes it. It sounds boring. It saves you days.

Agents need to know about the world, not just the content

One thing that hit me late: agents that produce content still need to verify it against reality.

You can't just generate, review, and publish. If your content references URLs, product features, or external resources, something in the pipeline needs to crawl, check, and validate that those things actually exist and are what you say they are. That's not a writing problem, it's an operations problem that ends up inside your pipeline whether you planned for it or not.

So the agent doing link injection became an agent that also runs verification. Cache management, staleness checks, what happens when a target has moved, all of that had to be designed. It added a layer I hadn't scoped.

The broader point: agentic systems that touch the real world (and most interesting ones do) inherit the complexity of the real world. Plan for it.

The model question is more interesting than you'd expect

I started with Sonnet 4.6 across the board. Fast, cost-efficient, handles orchestration and logic well.

But the writing quality wasn't where I needed it. The content that came out of Sonnet was competent. Opus 4.6 produced something noticeably better, more nuanced, better structure, stronger prose. For a pipeline where content quality is the whole point, that gap mattered.

So I moved the writing agent to Opus and kept everything else on Sonnet. Which immediately created a new problem: cost.

Opus isn't cheap at volume. When you're running a pipeline that processes multiple articles per day, with retries and revision loops, the token bill compounds fast. I had to get deliberate about when Opus actually gets invoked, only for first-draft generation and rewrites, not for every review pass or orchestration step that doesn't need that level of quality.

I also integrated OpenAI as a fallback for when Claude was taking too long on a given run. Not a permanent switch, just a safety valve to keep the pipeline moving. That added its own layer of routing logic: which model, when, based on what criteria.

What I didn't expect is how much of building agents ends up being model ops. You're not just building software logic, you're managing a portfolio of models with different cost profiles, speed characteristics, and quality outputs, and making routing decisions across them. That's a discipline in itself.

The edge cases are the real project

Here's something no one tells you: the initial build is maybe 30% of the work. The rest is edge cases.

What happens when an article is in the middle of a revision and the orchestrator tries to hand it to the editor? What happens when a verification step finds a stale link, does the whole pipeline stop, or does it warn and continue? What happens when the writer generates something that violates a content rule, retries, and violates it again?

Every one of those scenarios needs a decision. Warn or hard fail? Retry or escalate? Write to this field or that one? And the decisions interact, a retry policy in one agent affects the load on another.

After the initial build, I spent more time on these than on anything else. And honestly, that's where the interesting thinking happens. The pipeline logic is table stakes. The edge case handling is where you find out whether the system actually works in production.

What I'd do differently

  • Define the shared state contract first. Every field your agents read or write, documented before you build. This single thing would have saved me the most debugging time.
  • Scope model routing early. Decide which agents need the best model and which can run on something faster and cheaper. Don't default to one model everywhere and refactor later, the cost and latency implications show up fast at any real volume.
  • Expect the real world to intrude. If your pipeline touches live URLs, published content, or any external system, build the verification layer into the design from the start. It will be needed.
  • Exit codes are a contract. If your orchestrator stops everything on a non-zero exit, every agent needs to be deliberate about when it actually wants to stop things. Warn and fail are different states. Treat them differently.

The interesting thing about building agents for the first time is that the hard problems aren't really AI problems. They're distributed systems problems, state management, contracts between components, failure modes, cost optimization across heterogeneous services. The AI layer adds capability. The engineering layer determines whether that capability actually runs reliably.

If you're building something similar, what broke for you first?
